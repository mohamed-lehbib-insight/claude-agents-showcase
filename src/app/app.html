MULTI-SESSION

Use a Writer / Reviewer pattern

Beyond parallelising work, multiple sessions enable quality-focused workflows. A fresh context improves code review — Claude won't be biased toward code it just wrote. Spin up a second agent with no prior context to audit the first agent's output.

Long context → bias. Fresh context → honest review.
SESSION A — WRITER

Implement a rate limiter for our API endpoints

output →
SESSION B — REVIEWER

Review the rate limiter in @src/middleware/rateLimiter.ts. Look for edge cases, race conditions, and consistency with existing middleware patterns.

feedback →
SESSION A — REVISE

Here's the review feedback: [Session B output]. Address these issues.
